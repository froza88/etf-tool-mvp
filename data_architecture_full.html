🚨 第1章：核心问题 - 为什么数据总是不完整？

问题现象

2026-05-21 晚：year_3_return 覆盖率 79.9% (1174/1470)

2026-05-28 今：year_3_return 覆盖率 49.6% (734/1480)

👉 7天内丢失了 30.3% 的数据！

根本原因

pipeline.py 的 step_build() 函数使用"填表格"模式（全量重建）

每次运行 pipeline 的 build 步骤时，程序会：

  1. 从 数据源（可能是不完整的数据源）读取数据
  2. 创建一个 全新的空表格（dict）
  3. 把数据源的数据 填入新表格
  4. 用新表格 覆盖 etf_standard_data.json

结果：如果数据源缺少某个字段（比如 year_3_return），这个字段就会变成 null/0，之前积累的数据就丢失了！

❌ 第2章：错误模式 - "填表格"（全量重建）
🔴 填表格模式（当前错误）

比喻：每次做新的报表，都把旧报表扔掉，重新画一张空白表格，然后从（可能不完整的）数据源抄数据。

问题：

  • ❌ 数据源不完整 → 新表格也不完整
  • ❌ 之前积累的数据丢失
  • ❌ 每次都要重新获取所有数据（浪费API积分）
  • ❌ 数据质量无法累积提升

代码逻辑：

# step_build() 当前逻辑（错误） standard_etf = {} # ← 创建空表格 standard_etf['code'] = code standard_etf['year_3_return'] = calc.get('year_3_return') # ← 如果calc没有，就变成0 # ... 其他字段 json.dump(standard_etf) # ← 覆盖原文件
🟢 搭积木模式（正确方案）

比喻：保留旧报表，每次只添加新的积木块（缺失的数据），绝不扔掉已有的积木。

优势：

  • ✅ 已有数据永远保留
  • ✅ 只补充缺失的字段
  • ✅ 数据质量逐步提升
  • ✅ 节省API积分（只查缺失的）

代码逻辑：

# step_build() 应该的逻辑（正确） standard_data = json.load(existing_file) # ← 加载已有数据 for etf in standard_data: for key, value in new_data.items(): if key not in etf or etf[key] is None or etf[key] == 0: etf[key] = value # ← 只填充缺失/空字段 json.dump(standard_data) # ← 保存合并后的数据
✅ 第3章：正确模式 - "搭积木"（增量合并）

正确的数据存储逻辑应该是 "搭积木 + 填表格"混合模式：

🏗️ 混合模式逻辑

  1. 加载已有数据：从 etf_standard_data.json 读取现有数据（保留所有积木）
  2. 获取新数据：从数据源（AKShare/WeStock/Wind/盈米）查询最新数据
  3. 增量合并：只填充缺失/空的字段，不覆盖已有有效数据
  4. 保存：写回 etf_standard_data.json

🔄 数据流动路径

路径1（实时查询）：用户访问网页 → Flask读取 etf_standard_data.json → 展示数据

路径2（后台更新）：pipeline/build → 查询数据源 → 增量合并到 etf_standard_data.json

路径3（用户触发）：用户查看某只ETF → 后台查询该ETF最新数据 → 增量合并到本地缓存
